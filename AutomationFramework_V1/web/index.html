Overview

What is OneTest?

The OneTest is a PySpark-based data validation framework designed for automated testing with a plugin system for data transformations and file conversions. It provides a comprehensive solution for data quality testing and validation across different file formats.

Built with flexibility and ease of use in mind, this framework accommodates various testing needs from simple individual test runs to complex scheduled batch executions with detailed reporting.

  • Added support for reading scheduler configuration directly from test case YAML files
  • Simplified scheduling process by automatically reading frequency and other settings from YAML

Example YAML configuration:

scheduler:

  username: S123455

Core Components

The framework consists of several major components working together:

  • Data Validation System:
    • The validation module contains comprehensive tools for comparing datasets
    • Comparison.py serves as the primary entry point for validation operations
    • Supports multiple validation types: count validation, schema validation, data validation, rule validation
  • Modular Converter System:
    • Versioned converter modules in the converters directory
    • Each converter follows a standard interface with a convertToSparkDataFrame function
    • Supports various file formats (CSV, PSV, XML, JSON, Mainframe/EBCDIC, etc.)
  • Plugin Architecture:
    • core/plugin_integration.py and plugin_runner.py provide a flexible plugin system
    • Allows dynamic loading and execution of transformation plugins
    • Supports both data processing plugins and file converter plugins
  • Test Runner & Visualization:
    • run_tests_visual.py provides a visual interface for test execution
    • Shows stage-by-stage test execution with flow diagrams and color-coded status indicators
    • Organizes test execution into logical stages for better monitoring
  • Test Scheduler:
    • Allows scheduling of test cases with configurable frequencies
    • Supports continuous execution mode that periodically checks for tests
  • SQLite Database: - Stores test definitions, execution results, and scheduling information

Architecture Highlights

  • Modular Design: The framework follows a modular approach with clear separation of concerns between validation, conversion, and execution components.
  • Extensible Plugin System: The plugin architecture enables easy extension without modifying core code.
  • Versioned Components: Converters and other modules use versioning to ensure backward compatibility.
  • Comprehensive Validation: Multiple validation types (count, schema, data, rules) provide thorough testing capabilities.
  • Visual Feedback: The visual test runner provides real-time progress visualization.
  • Database Integration: Test results are stored in a database for tracking and reporting.
  • YAML-Based Configuration: Test cases are defined in YAML format, allowing for declarative test specification.

Quick Start Guide

Test Scheduler

The framework includes a powerful scheduler system that allows test cases to be scheduled for automatic execution at specified intervals.

Adding Tests to Scheduler

There are three ways to add test cases to the scheduler:

  1. Through the interactive menu (Option 5: Add Test to Scheduler)
  2. Through the interactive menu (Option 6: Add Test to Scheduler via YAML)
  3. Direct command: python3 add_to_scheduler.py <yaml_file>

YAML Configuration

When using YAML files for scheduling, include a scheduler section in your test case YAML:

scheduler:
  enabled: 'yes'
  frequency: daily  # Options: daily, weekly, monthly
  username: S123455

Key Features

  • Support for daily, weekly, and monthly scheduling
  • Automatic tracking of execution times
  • Retry mechanism for failed tests
  • Detailed execution logging
  • YAML-based configuration

Running the Scheduler

Start the scheduler in continuous mode:

python3 run_schedule_test_cases.py --continuous --interval 600

Getting Started in 5 Minutes

New to OneTest? Follow these steps to get up and running quickly:

  1. Setup Environment 
    # Clone the repository
git clone https://github.com/yourusername/AutomationFramework_V1.git
cd AutomationFramework_V1

# Install dependencies
pip install -r requirements.txt
  2. Use the Interactive Interface

    The easiest way to get started is using our interactive menu:

    # Launch the interactive menu
python3 run_interactive.py

    This provides a menu-driven interface to all major framework functions.

  3. Run Your First Test 
    # Run a sample test case
python3 run_test_cases.py Testcases/TC_001.yaml
  4. Create Your Own Test

    Create a new YAML file in the Testcases directory with your test configuration:

    test_case_name: "My First Test"
sid: "my_source_id"
table_name: "my_table"

source_configuration:
  source_type: "csv"
  source_location: "data/source.csv"
  
target_configuration:
  target_type: "csv"
  target_location: "data/target.csv"
  
validation_types:
  - "count"
  - "schema"
  - "data"
  5. Schedule Your Test 
    # Add to scheduler with daily frequency
python3 add_to_scheduler.py my_test.yaml --frequency daily
  6. Run Scheduled Tests 
    # Execute all scheduled tests
python3 run_scheduler.py --run-now

Unique Strengths

Unique Strengths of OneTest

Based on a review of the codebase, this automation framework has several distinctive advantages compared to market alternatives:

  1. PySpark-Native Visual Test Runner

    The framework offers a unique terminal-based visual test runner with real-time flow diagrams showing test execution stages. Unlike many commercial tools that require heavy UIs, this provides an elegant visual experience directly in the terminal, making it accessible in environments where GUI applications aren't practical.

  2. Versioned Converter System

    The modular, versioned data loader system (format_versions.yaml with converter modules like csv2sparkdf_v1_1_2.py) is a particularly elegant design. This allows for precise control over file format handling while maintaining backward compatibility - a feature often missing in commercial tools that force users to upgrade all components simultaneously.

  3. Specialized File Format Support

    The framework shows particular strength in handling specialized file formats like:

    • Mainframe/EBCDIC files
    • Episodic data
    • Fixed-width formats

    Many commercial tools require expensive add-ons or lack proper support for these legacy formats.

  4. Lightweight Plugin Architecture

    The plugin system strikes an excellent balance between flexibility and simplicity. Unlike the often over-engineered plugin systems in commercial tools, this framework follows clear conventions while remaining lightweight and easy to extend.

  5. Balanced Validation Approach

    The framework offers a balanced combination of validation types (count, schema, data, rule) in a unified system. Commercial tools often separate these capabilities into different modules with inconsistent interfaces.

  6. Test Case Scheduler with Visual Monitoring

    The combination of scheduler capabilities with visual progress tracking creates a powerful monitoring system that many commercial tools lack. The ability to schedule tests with different frequencies while visually tracking their execution is particularly valuable.

  7. Deep PySpark Integration

    The framework leverages PySpark's capabilities more deeply than most commercial alternatives, which often treat Spark as just another connector rather than a core processing engine.

  8. Domain-Specific Testing Language

    The YAML-based test definition approach creates a domain-specific language for data validation that's more accessible than the proprietary scripting languages used by many commercial tools.

  9. Self-Contained Architecture

    Unlike many cloud-dependent commercial solutions, this framework operates as a self-contained system, enabling deployment in secure environments with limited external connectivity.

These distinctive capabilities position this framework uniquely in the market, particularly for organizations with specialized data validation needs, legacy data formats, or requirements for a lightweight yet powerful testing framework.

Main Data Flows

  1. Test Execution Flow: User initiates test through UI → Test Case Runner loads YAML configuration → Validator executes tests → Results stored in DB → Reports generated
  2. Scheduled Test Flow: Scheduler checks for due tests → Executes tests via Test Case Runner → Results stored with schedule ID → Updates schedule status → Reports generated
  3. Data Validation Flow: Validation Engine loads source and target data → Performs validations (schema, count, data, rules) → Generates comparison results → Stores results in DB
  4. File Conversion Flow: Data Source identifies file type → File Converter applies transformation → Converted data passed to Validation Engine
  5. Parallel Execution Flow: Multiple test cases are executed simultaneously → ThreadPool manages concurrent execution → Results are consolidated → Overall summary is generated
  6. Advanced Logging Flow: Execution mode (--silent/--quiet) sets log level → Logger filters output based on mode → Only relevant information is displayed → Full logs still stored in database
  7. Interactive Command Flow: User selects operation → Interactive UI presents options → Command executed → Results displayed → Return to menu

Database

Database Structure Overview

The framework uses an SQLite database to store test definitions, execution results, and schedule information. The database follows normalization principles to minimize redundancy while maintaining data integrity.

Table Name Primary Key Purpose Key Fields
testcase traceability_id Central table that stores all test case definitions and metadata test_case_name, sid, table_name, source_type, target_type, validation_types
testcase_execution_stats execution_id Stores detailed results of test case executions traceability_id (FK), execution_run_id (FK), status, source_count, target_count, execution_start/end_time
execution_run execution_run_id Tracks batch execution runs and summary statistics start_time, end_time, total_test_cases, passed_test_cases, failed_test_cases, overall_status
execution_log log_entry_id Stores detailed logs for each test execution step execution_id (implicit FK), traceability_id (FK), execution_run_id (FK), timestamp, level, message
scheduler schedule_id Maintains test schedules with frequency and timing information traceability_id (FK), yaml_file_path, frequency, day_of_week, day_of_month, next_run_time, last_run_time

Key Relationships Explained

  1. Test Case Definition to Execution (1:Many): A single test case (testcase.traceability_id) can have multiple executions (testcase_execution_stats.traceability_id). Each execution record is tied to exactly one test case definition.
  2. Test Case Execution to Logs (1:Many): A single test execution (testcase_execution_stats.execution_id) can have multiple log entries (execution_log.execution_id). Each log entry is associated with exactly one test execution.
  3. Batch Run to Test Executions (1:Many): A batch run (execution_run.execution_run_id) can include multiple test executions (testcase_execution_stats.execution_run_id). Each test execution can belong to at most one batch run.
  4. Test Case Definition to Schedule (1:Many): A test case (testcase.traceability_id) can have multiple schedules (scheduler.traceability_id). Each schedule is tied to exactly one test case.
  5. Batch Run to Logs (1:Many): A batch run (execution_run.execution_run_id) can have multiple log entries (execution_log.execution_run_id). Log entries can be associated with both individual test executions and batch runs.

Usage Guide

Using the Interactive Interface

The simplest way to use the framework is through the interactive menu:

# Launch the interactive menu
python3 run_interactive.py

# Start directly in scheduler mode
python3 run_interactive.py --mode scheduler

# Start directly in test execution mode
python3 run_interactive.py --mode test

# Start directly in visual test mode
python3 run_interactive.py --mode visual

Command Line Usage

Basic Command (Most Common)

# Run all tests in the default Testcases directory
python3 run_test_cases.py

This is the most frequently used command that executes all test cases in the default directory.

Advanced Test Running Options

# Run a single test
python3 run_test_cases.py Testcases/TC_001.yaml

# Run all tests in a specific directory
python3 run_test_cases.py --test-dir Testcases

# Run with pattern matching
python3 run_test_cases.py --pattern "TC_00*.yaml"

Advanced Execution Options

# Silent mode - suppresses INFO level logs
python3 run_test_cases.py --silent

# Quiet mode - shows only test name and status
python3 run_test_cases.py --quiet

# Parallel execution with default (2 workers)
python3 run_test_cases.py --parallel

# Parallel execution with custom number of workers
python3 run_test_cases.py --parallel 4

# Combining options
python3 run_test_cases.py --quiet --parallel 3
python3 run_test_cases.py --silent --parallel
python3 run_test_cases.py --test-dir CustomTests --quiet --parallel 5

Managing Schedules

# Add test to scheduler
python3 add_to_scheduler.py Testcases/TC_001.yaml --frequency daily

python3 run_tests_visual.py --test-file Testcases/TC_001.yaml Runs a specific test case using the visual test runner, showing detailed progress through each stage of execution.

Best Practices

  • Use the run_interactive.py script for a guided, menu-driven approach to the framework
  • Check framework version with python3 version_info.py when troubleshooting or reporting issues
  • Use the run_tests_visual.py script during development for better visibility into test execution stages
  • Create a cron job or scheduled task to run python3 run_scheduler.py periodically (e.g. every hour) to automatically execute scheduled tests
  • For batch test execution, use run_test_cases.py without arguments to run all tests in sequence
  • Use descriptive test case names in YAML files to make the test results and reports more informative
  • When adding tests to the scheduler, consider spreading out test execution times to avoid resource contention
  • Regularly back up your database with python3 db_utilities.py --backup
  • Run clean.py periodically to prevent disk space issues from log files and temporary artifacts
  • When debugging test failures, use --debug mode for more detailed logging output

How do I install the framework?

Clone the repository and install dependencies:

git clone https://github.com/username/AutomationFramework_V1.git
cd AutomationFramework_V1
pip install -r requirements.txt

How do I create my first test?

Create a YAML file in the Testcases directory with source and target configurations.

See the Quick Start Guide for a basic example.

Can I schedule tests to run automatically?

Yes, use add_to_scheduler.py to schedule tests with daily, weekly, or monthly frequency:

python3 add_to_scheduler.py Testcases/TC_001.yaml --frequency daily

Then run the scheduler in continuous mode:

python3 run_scheduler.py --continuous

How do I use the interactive mode?

Run python3 run_interactive.py to launch the interactive interface.

This provides a menu-driven approach to all framework functions without needing to remember command-line arguments.

How-Tos

Common Tasks

How to Run Tests with Visual Feedback

Use the visual test runner for real-time execution monitoring:

python3 run_tests_visual.py --test-file Testcases/TC_001.yaml

How to Add Tests to the Scheduler

Schedule a test to run at regular intervals:

# Daily at 9 AM
python3 add_to_scheduler.py Testcases/TC_001.yaml --frequency daily --time "09:00"

# Weekly on Monday
python3 add_to_scheduler.py Testcases/TC_001.yaml --frequency weekly --day-of-week 1

How to Convert Files

Convert between different file formats:

# CSV to PSV conversion
python3 run_file_converter.py --config csv_to_psv --input data.csv --output data.psv

# JSON to CSV conversion
python3 run_file_converter.py --config json_to_csv --input data.json --output data.csv

Version Fallback System

The framework now includes a robust version fallback mechanism that automatically handles failures in beta versions by falling back to stable versions. This system provides comprehensive tracking of all fallback decisions and module execution issues.

Key Features

  • Automatic fallback from beta to stable versions when errors occur
  • Comprehensive tracking of all version fallbacks
  • Detailed error logging for module execution issues
  • Integration with the main test execution flow

Database Schema

Version Fallback Decisions Table

Column Type Description
fallback_id INTEGER Primary key for the fallback decision
execution_id TEXT ID of the test execution
test_case_id TEXT ID of the test case
traceability_id TEXT Traceability ID for the test
module_type TEXT Type of module (converter, processor, etc.)
operation_type TEXT Type of operation being performed
initial_version TEXT Initial version that failed
fallback_version TEXT Version used as fallback
error_message TEXT Error message from the failure
successful BOOLEAN Whether the fallback was successful
timestamp TIMESTAMP When the fallback occurred

Module Execution Issues Table

Column Type Description
issue_id INTEGER Primary key for the issue
execution_id TEXT ID of the test execution
test_case_id TEXT ID of the test case
module_name TEXT Name of the failing module
module_type TEXT Type of module
attempted_version TEXT Version that failed
framework_version TEXT Framework version in use
error_message TEXT Detailed error message
stack_trace TEXT Full error stack trace
timestamp TIMESTAMP When the issue occurred

Configuration Example


formats:
  csv:
    current_version: 1.1.2
    current_version_type: beta
    stable_version: 1.1.1
    available_versions:
      - version: 1.1.1
        type: stable
        release_date: "2023-12-01"
      - version: 1.1.2
        type: beta
        release_date: "2024-01-15"

How It Works

  1. When a module is loaded, the system first checks the format_versions.yaml file
  2. If the current version is beta, the system:
    • Attempts to use the beta version first
    • If it fails, logs the failure in module_execution_issues
    • Automatically falls back to the stable version
    • Records the fallback decision in version_fallback_decisions
  3. The fallback process is transparent to the test execution
  4. All decisions and issues are tracked for analysis

Monitoring and Analysis

The framework provides several ways to monitor and analyze version fallbacks:

  • Query the version_fallback_decisions table to track fallback patterns
  • Analyze module_execution_issues to identify problematic versions
  • Use the execution logs to understand the full context of failures
  • Generate reports on version stability and fallback frequency
